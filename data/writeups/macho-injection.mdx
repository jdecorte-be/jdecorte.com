## Understanding Mach-O: The Blueprint of macOS Binaries

If you've ever wondered what's actually happening beneath the surface when macOS runs a program, Mach-O files are your answer. They're the executable format powering every native macOS and iOS application — and once you understand their structure, you'll see the entire system in a completely different light.

This guide walks you through the anatomy of Mach-O binaries, then gets into the advanced stuff: dynamic library injection, entry point manipulation, and the kind of low-level tricks that make a real packer tick. By the end, you'll be comfortable not just reading these files, but bending them to your will.

Let's get into it.

---

## What Is a Mach-O File, Really?

Strip away the abstraction and a Mach-O file is just a precisely structured sequence of bytes — but those bytes carry everything macOS needs to:

- Load the program into the right memory regions
- Resolve external library dependencies at runtime
- Enforce memory permissions (read, write, execute)
- Know exactly where to begin execution

The format is modular by design. Every section has a defined role, every offset is deliberate, and the kernel reads it all before a single line of your code ever runs. That precision is what makes Mach-O files so powerful to work with — and so unforgiving when you get something wrong.

## The Anatomy of a Mach-O Binary

### 1. The Header — First Bytes, First Impressions

The Mach-O header sits at byte zero and is the first thing the kernel reads. It's compact but critical — corrupting it means the binary doesn't even get off the ground.

Key fields you'll encounter:

| Field | What It Tells the System |
|---|---|
| `magic` | Identifies the file as Mach-O (`0xFEEDFACF` for 64-bit) |
| `cputype` / `cpusubtype` | Target architecture — ARM64 for Apple Silicon |
| `filetype` | Executable, dylib, object file, etc. |
| `ncmds` | Number of load commands that follow |
| `sizeofcmds` | Total byte size of all load commands |
| `flags` | Behavior flags (PIE, dyldlink, etc.) |

Want to inspect a real header? Run:

```bash
otool -h /path/to/binary
```

You'll immediately see the architecture and file type staring back at you.

---

### 2. Load Commands — The Real Instructions

Immediately following the header, load commands are where the actual orchestration happens. These tell the dynamic linker and kernel how to set up the process before execution begins.

Some of the most important ones:

- **`LC_SEGMENT_64`** — Maps a segment of the binary into virtual memory
- **`LC_LOAD_DYLIB`** — Declares a dynamic library dependency
- **`LC_MAIN`** — Specifies the entry point offset
- **`LC_DYLD_INFO_ONLY`** — Contains binding and rebase information for dyld
- **`LC_CODE_SIGNATURE`** — Points to the cryptographic signature blob

This is where most binary modification work lives. Adding a library? Insert an `LC_LOAD_DYLIB`. Redirecting execution? Patch `LC_MAIN`. Understanding load commands is non-negotiable for Mach-O work.

List all load commands in a binary with:

```bash
otool -l /path/to/binary
```

---

### 3. Segments and Sections — The Memory Layout

Segments define large regions of the binary that get mapped into memory. Sections live inside segments and hold specific types of data.

| Segment | Role | Notable Sections |
|---|---|---|
| `__TEXT` | Read-only, executable code | `__text` (machine code), `__stubs`, `__unwind_info` |
| `__DATA` | Readable and writable runtime data | `__data`, `__bss`, `__got`, `__la_symbol_ptr` |
| `__LINKEDIT` | Linker metadata | Symbol tables, string tables, code signature |

When you're injecting code, you're typically either appending to `__TEXT`, adding a new segment, or — in more aggressive approaches — squeezing into the padding between sections. The permission model here matters enormously: memory that isn't marked executable will trigger an immediate crash or kernel exception on modern Apple Silicon hardware.

---

### 4. The Entry Point — Where Execution Begins

The `LC_MAIN` load command holds a single critical value: the file offset to the `main()` function. When dyld finishes setting up the process, this is where control transfers.

For a packer, this is your primary hook. The classic pattern looks like this:

1. Save the original `LC_MAIN` offset
2. Redirect the entry point to your custom stub
3. The stub performs its work — decrypting, unpacking, whatever you need
4. Execution jumps to the original entry point and the program runs normally

Everything else in the packer design flows from this single redirection.

## Advanced Mach-O: Dynamic Library Injection

With the structure clear in your head, let's get into the real meat of this project — dynamically injecting a library into an existing Mach-O binary.

The goal: insert an `LC_LOAD_DYLIB` command that causes macOS to load your library at runtime, every time the target binary runs. No source code needed. No recompilation. Just raw binary surgery.

---

### How Dynamic Library Injection Works

When dyld processes a Mach-O binary, it walks every load command sequentially. When it hits an `LC_LOAD_DYLIB`, it loads that library before handing off to `LC_MAIN`. Your injection exploits this behavior by appending a new `LC_LOAD_DYLIB` entry that points to a library you control.

The library loads, your code runs, and the original program continues unaffected — or modified, depending on what your library does.

---

### Step 1: Understanding the `LC_LOAD_DYLIB` Structure

Before writing anything, know exactly what you're inserting:

```c
struct dylib_command {
    uint32_t cmd;               // LC_LOAD_DYLIB = 0xC
    uint32_t cmdsize;           // Total size: struct + path string + padding
    struct dylib {
        uint32_t name;          // Offset to path string, relative to struct start
        uint32_t timestamp;     // Usually set to 2 (build time placeholder)
        uint32_t current_version;
        uint32_t compatibility_version;
    } dylib;
    // Library path string follows immediately, null-terminated + padded to 8 bytes
};
```

The `cmdsize` field must account for the struct itself, the path string, and any padding required to maintain 8-byte alignment. Getting this wrong corrupts every load command that follows.

---

### Step 2: Finding Space for the New Command

Here's the first real challenge. Load commands live between the header and the first segment's content. If there's no free space there — and often there isn't — you have a few options:

- **Use existing padding** — Many binaries have a small gap between the end of load commands and the start of `__TEXT` content. Check `sizeofcmds` against the segment's file offset.
- **Shift file content** — More invasive; requires updating every file offset in every segment and section.
- **Extend `__LINKEDIT`** — Append data to the end of the file, where `__LINKEDIT` lives, and adjust its size accordingly.

For a first implementation, checking for existing padding is the cleanest approach.

---

### Step 3: Writing the Command

Once you've confirmed available space, write the struct and path:

```c
// Build the command in memory first
struct dylib_command new_cmd = {0};
new_cmd.cmd = LC_LOAD_DYLIB;
new_cmd.dylib.name = sizeof(struct dylib_command); // Path starts right after struct
new_cmd.dylib.timestamp = 2;
new_cmd.dylib.current_version = 0x10000;       // 1.0.0
new_cmd.dylib.compatibility_version = 0x10000; // 1.0.0

const char *lib_path = "/usr/local/lib/my_library.dylib";
size_t path_len = strlen(lib_path) + 1;

// Calculate aligned cmdsize
uint32_t raw_size = sizeof(struct dylib_command) + path_len;
uint32_t aligned_size = (raw_size + 7) & ~7; // Round up to 8-byte boundary
new_cmd.cmdsize = aligned_size;

// Seek to insertion point and write
fseek(binary, insert_offset, SEEK_SET);
fwrite(&new_cmd, sizeof(struct dylib_command), 1, binary);
fwrite(lib_path, path_len, 1, binary);

// Write alignment padding
uint8_t padding[8] = {0};
fwrite(padding, aligned_size - raw_size, 1, binary);
```

---

### Step 4: Updating the Header

The header must reflect your changes or dyld will ignore or misparse your new command:

```c
// Read existing header
struct mach_header_64 header;
fseek(binary, 0, SEEK_SET);
fread(&header, sizeof(header), 1, binary);

// Update command count and total size
header.ncmds += 1;
header.sizeofcmds += aligned_size;

// Write it back
fseek(binary, 0, SEEK_SET);
fwrite(&header, sizeof(header), 1, binary);
```

Two fields. Easy to forget one of them. Don't.

---

### Step 5: Handling Code Signatures

Any modification to a signed binary invalidates its signature. macOS will refuse to run it. For testing and development, you have two paths:

**Option A — Remove the signature entirely:**

```bash
codesign --remove-signature ./modified_binary
```

Simplest option. Works on binaries that don't require hardened runtime.

**Option B — Re-sign after modification:**

```bash
codesign -s - ./modified_binary  # Ad-hoc signature (local testing only)
```

For anything beyond personal testing, you'll need a valid Apple Developer identity. Also note that on ARM64 Apple Silicon, SIP and the kernel's trust chain add additional constraints that ad-hoc signing doesn't always satisfy. Know your environment.

---

### Step 6: Verification

Confirm your injection landed correctly:

```bash
# Should show your injected library in the dependency list
otool -L ./modified_binary

# Inspect all load commands and verify ncmds / sizeofcmds updated correctly
otool -l ./modified_binary | head -30
```

If you see your library path in the output and the binary runs — congratulations, you've performed surgical Mach-O modification by hand.

## Common Failure Modes

If you're seeing crashes or load failures, run through this checklist:

| Problem | Likely Cause |
|---|---|
| `killed: 9` on launch | Code signature invalid — re-sign or strip it |
| `dyld: malformed mach-o` | `ncmds` or `sizeofcmds` mismatch, or bad alignment |
| Library not loading | Path incorrect or library missing at runtime |
| Segfault at entry | Entry point offset wrong after modification |
| `SIGBUS` immediately | Segment file offsets misaligned after content shift |

When debugging, `lldb` and `DYLD_PRINT_LIBRARIES=1` are your best friends:

```bash
DYLD_PRINT_LIBRARIES=1 ./modified_binary
```

This prints every library dyld attempts to load, letting you confirm your injected dylib is being picked up — or diagnose exactly where it's failing.

## Putting It All Together

The full workflow for a dynamic library injection:

1. **Parse the header** — Read `ncmds`, `sizeofcmds`, confirm architecture
2. **Walk load commands** — Map out existing commands, find available space
3. **Build the `LC_LOAD_DYLIB` struct** — Set all fields, calculate aligned `cmdsize`
4. **Write the command and path** — Insert at the correct offset with proper padding
5. **Update the header** — Increment `ncmds`, add to `sizeofcmds`
6. **Handle code signing** — Strip or re-sign as appropriate for your environment
7. **Verify with `otool`** — Confirm the injection appears in load commands
8. **Test execution** — Run the binary, check library loads, debug with `lldb` if needed

This same pattern — read, modify, update bookkeeping, verify — applies to virtually every Mach-O modification you'll ever do. Master it here and the more complex stuff (segment injection, entry point redirection, packing/unpacking stubs) becomes a natural extension.
